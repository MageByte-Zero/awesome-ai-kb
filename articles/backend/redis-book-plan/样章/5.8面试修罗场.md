## 5.8 面试修罗场

这一章节，从面试角度把 Redis 实战技巧、性能优化、分布式锁、数据一致性解决方案高频知识点提炼，带你融会贯通梳理知识点。

### 5.8.1 Redis 性能怎么调优？

> 吴彦祖面试官：“Redis 性能突然变慢卡顿如何做优化？”

在优化之前我们先确定 Redis 实例的性能基线，用于判断是否真的变慢。redis-cli 可执行脚本提供了 `–intrinsic-latency` 选项，用来监测和统计测试期间内的最大延迟（以毫秒为单位），这个延迟可以作为 Redis 的基线性能。

此外，还可以使用慢日志功能定位到那些超出指定执行时间的慢命令，默认情况下命令若是执行时间超过 10ms 就会被记录到日志。

以上是被动解决方式，我们主动出击使用以下清单对 Redis 做性能优化。

1. 使用 Redis pipline 将多个命令连接在一起减少网络往返次数。
2. 针对慢指令优化：在 Cluster 集群中，将聚合运算等 O(N) 时间复杂度操作放到 slave 上运行或者在客户端完成。使用更高效的命令代替。比如使用增量迭代的方式，避免一次查询大量数据，具体请查看 `SCAN、SSCAN、HSCAN、ZSCAN`命令。
3. 禁用 Linux 内存大页：采用了内存大页，生成 RDB 期间即使客户端修改的数据只有 50B 的数据，Redis 可能需要复制 2MB 的大页。当写的指令比较多的时候就会导致大量的拷贝，导致性能变慢。
4. 排查是否出现 swap 交换，增加机器内存、或者 Cluster 节点数量。
5. 合理设置持久化，我们通常只是将 Redis 用于缓存，数据未命中从数据获取，并不需要很高的数据可靠性，建议设置成 no 或者 everysec。
6. 控制实例的内存大小：我们通常只是将 Redis 用于缓存，数据未命中从数据获取，并不需要很高的数据可靠性，建议设置成 no 或者 everysec。
7. 避免大量的 Key 在相同的时间过期。
8. 防止出现 bigkey： Redis 中那些存有较大量元素的集合或列表、大对象的字符串占用较大内存空间的键值对数据称为 Bigkey。 bigkey 会造成以下问题。
   1. 大键会占用大量的内存，可能导致 Redis 实例的内存使用率过高。
   2. 在进行持久化操作（如 RDB 快照、AOF 日志）延迟过高。
   3. 网络传输的压力增大。

### 5.8.2 Redis 分布式锁实现原理

> 面试官：“分布式锁有哪些特性？”

- **互斥性（Mutual Exclusion）：** 任意时刻只有一个客户端能够持有锁。
- **有限等待（Finite Wait）**：分布式锁应该具有合理的超时机制，确保在一定时间内获取锁，避免死锁。
- **安全性（Safety）**：加锁和解锁必须是同一个客户端（线程），客户端自己不能把别人加的锁给解了。
- **高可用性（High Availability）：** 分布式锁的实现应该具备高可用性，确保在节点故障或网络分区的情况下，系统仍能正常工作。

> 面试官：“如何实现分布式锁？”

使用 `SET lockKey randomValue NX PX expireTime`实现。

- `lockKey` 表示锁的资源，value 设置成 1。
- `NX`：表示只有 `lockKey` 不存在的时候才能 `SET` 成功，从而保证只有一个客户端可以获得锁。
- `PX expireTime`设置锁的超时时间，单位是毫秒；也可以使用 `EX seconds`以秒为单位设置超时时间。

特别需要注意的是，客户端加锁时设置一个“唯一标识” ，释放锁时判断锁的唯一标识与客户端的标识是否匹配，匹配才能删除。

> 面试官：“可重入锁如何实现？”

可以使用 Redis hash 结构实现，key 表示被锁的共享资源， hash 结构的 fieldKey 存储客户端唯一标识，fieldKey 的 value 则保存加锁的次数。

> 面试官：“业务逻辑的执行时间超过设置的超时时间导致锁被提前解锁了怎么办？”

这个时间不能瞎写，一般要根据在测试环境多次测试，然后压测多轮之后，比如计算出接口平均执行时间 200 ms。那么锁的**超时时间就放大为平均执行时间的 3~5 倍。**

> 面试官：“谢霸哥：“有没有完美的方案呢？不管时间怎么设置都不大合适。””

我们可以给获得锁的线程开启一个**守护线程**，用来给当前客户端快要过期的锁续航，续命的前提是，得判断是不是当前进程持有的锁，如果不是就不进行续。

**如果快要过期，但是业务逻辑还没执行完成，自动对这个锁进行续期，重新设置过期时间。**

Redisson 框架就海鲜了这个机制，称之为看门狗。他的执行原理就是在加锁成功后异步执行一个定时任务，该定时任务使用了 Netty 的时间轮来实现。

> 面试官：“主从复制模式下，分布式锁会有什么问题？”

主从架构数据同步复制实现的数据同步是异步的。可能会出现在 master 获取的分布式锁成功，此时出现宕机，数据还没复制到 slave 就会出现锁丢失的情况。

另外，如果我们没有设置 Redis 的持久化，也会出现上面的问题。启用 AOF 持久化，情况将会改善很多。

### 5.8.3 Redis 与 MySQL 数据一致性如何解决？

> 面试官：“缓存使用策略有哪些？”

缓存使用策略包括 Cache-Aside（旁路缓存）、Read/Write Through （读写穿透）和 Write-Behind（异步缓存写入）。

**选择适当的缓存使用策略取决于应用程序的需求和性能目标**。例如，Cache-Aside 适用于读多写少的场景，Read Through 和 Write Through 通常配合一起使用，适用于需要自动加载和同步的场景，而 Write-Behind 适用于需要延迟写入和异步刷新的场景。

> 面试官：“数据库与缓存一致性解决方案有哪些？”

**在说解决方案之前，先达成一个共识，缓存是通过牺牲强一致性来提高性能的。**

这是由**CAP 理论**决定的。缓存系统适用的场景就是非强一致性的场景，它属于`CAP`中的`AP`，如果需要数据库和缓存数据保持强一致，就不适合使用缓存。

**针对 Cache-Aside (旁路缓存) 策略，写操作使用先更新数据库，再删除缓存**的情况下，我们来分析下数据一致性解决方案都有哪些？

- 延时双删策略。
- 删除缓存重试机制。
- 读取 binlog 异步删除缓存。
